# 04 - 状态管理与权限系统

> Claude Code 源码讲义 -- 状态管理、Hook 系统与权限控制流程

---

## 一、全局状态管理

### 1.1 自研 Store（非 Redux/Zustand）

核心实现在 `src/state/store.ts`，仅 30 行，极简发布-订阅模式：

```
createStore<T>(initialState, onChange?) => { getState, setState, subscribe }
```

- **setState** 接收 `(prev) => next` 更新函数，通过 `Object.is` 跳过无变更
- **subscribe** 返回 unsubscribe 函数，标准 listener 模式
- **onChange** 可选回调，在 state 变更时触发副作用（用于同步 CCR、持久化配置等）

### 1.2 AppState 结构（`src/state/AppStateStore.ts`）

AppState 是一个巨大的类型定义，核心字段分组：

| 分组 | 关键字段 | 说明 |
|------|---------|------|
| 基础配置 | `settings`, `verbose`, `mainLoopModel` | 用户设置、模型选择 |
| 权限 | `toolPermissionContext` | 权限模式、allow/deny 规则列表 |
| MCP | `mcp.clients/tools/commands` | MCP 服务器连接与工具注册 |
| 插件 | `plugins.enabled/disabled/errors` | 插件加载状态 |
| 任务 | `tasks`, `foregroundedTaskId` | 后台任务（Agent 子任务） |
| 团队 | `teamContext`, `inbox` | Swarm 多 agent 协作 |
| 投机执行 | `speculation` | 投机执行状态机 |
| Bridge | `replBridgeEnabled/Connected/...` | 与 claude.ai 的双向桥接 |
| 拒绝追踪 | `denialTracking` | auto 模式下连续拒绝计数 |

### 1.3 React 集成（`src/state/AppState.tsx`）

- `AppStateProvider` 创建 Store 实例，通过 React Context 注入
- `useAppState(selector)` -- 基于 `useSyncExternalStore`，按 selector 订阅切片，避免全量重渲染
- `useSetAppState()` -- 仅获取 setState，组件不订阅任何状态变化
- `useAppStateMaybeOutsideOfProvider()` -- 安全版本，Provider 外返回 undefined

### 1.4 状态变更副作用（`src/state/onChangeAppState.ts`）

通过 Store 的 `onChange` 回调集中处理副作用：

- **权限模式变更** -> 通知 CCR (`notifySessionMetadataChanged`) + SDK (`notifyPermissionModeChanged`)
- **模型变更** -> 同步到 settings 文件 + 全局覆盖
- **设置变更** -> 清除 API key / AWS / GCP 凭证缓存，重新加载环境变量

## 二、权限模式（Permission Mode）

定义在 `src/utils/permissions/PermissionMode.ts`，共 6 种模式：

| 模式 | 含义 | 行为 |
|------|------|------|
| `default` | 默认模式 | 每次工具调用需用户确认 |
| `plan` | 计划模式 | 仅允许只读操作，写操作需确认 |
| `acceptEdits` | 接受编辑 | 自动允许文件编辑，其他仍需确认 |
| `bypassPermissions` | 绕过权限 | 自动允许所有操作（危险） |
| `dontAsk` | 不询问 | 将所有 `ask` 转为 `deny`（自动拒绝） |
| `auto` | 自动模式 | 使用 AI 分类器判断是否允许（内部功能） |

用户可通过 Shift+Tab 在模式间循环切换。

## 三、权限规则系统

### 3.1 规则类型（`src/utils/permissions/PermissionRule.ts`）

```
PermissionRule = { source, ruleBehavior, ruleValue: { toolName, ruleContent? } }
```

- **ruleBehavior**: `allow` | `deny` | `ask`
- **ruleValue**: 工具名 + 可选内容（如 `Bash(prefix:git *)` 中的 `prefix:git *`）
- **source**: 规则来源 -- `userSettings` / `projectSettings` / `managedSettings` / `cliArg` / `session` 等

### 3.2 规则匹配（`src/utils/permissions/permissions.ts`）

三类规则列表存储在 `ToolPermissionContext` 中：
- `alwaysAllowRules[source]` -- 自动允许
- `alwaysDenyRules[source]` -- 自动拒绝
- `alwaysAskRules[source]` -- 强制询问

匹配逻辑：
1. 精确匹配工具名（如 `Edit`）
2. MCP 服务器级匹配（如 `mcp__server1` 匹配该服务器所有工具）
3. 带内容的规则通过工具自身的 `checkPermissions()` 判断（如 Bash 的前缀匹配）

## 四、权限判定流程

### 4.1 总入口：`hasPermissionsToUseTool()`

位于 `src/utils/permissions/permissions.ts`，判定流程：

```
1. 工具自身的 checkPermissions() 返回 allow/deny/ask
2. 如果 allow -> 直接通过
3. 如果 deny -> 直接拒绝
4. 如果 ask -> 根据当前权限模式做后处理：
   a. dontAsk 模式 -> 转为 deny
   b. auto 模式 -> 调用 AI 分类器判断
   c. 其他 -> 返回 ask 给上层处理
```

### 4.2 Hook 入口：`useCanUseTool()`

位于 `src/hooks/useCanUseTool.tsx`，是 React 层的权限调用入口：

```
useCanUseTool(tool, input, context, message, toolUseID)
  |
  +-- createPermissionContext()   // 构建权限上下文
  +-- hasPermissionsToUseTool()   // 核心权限判定
  |
  +-- 结果为 allow -> 直接返回
  +-- 结果为 deny  -> 记录日志，返回拒绝
  +-- 结果为 ask   -> 进入交互流程：
       |
       +-- handleCoordinatorPermission()  // 协调器模式
       +-- handleSwarmWorkerPermission()  // Swarm 工作者模式
       +-- handleInteractivePermission()  // 交互模式（主流程）
```

## 五、权限处理器（Handler）

### 5.1 交互处理器（`interactiveHandler.ts`）

主 agent 的权限对话框流程。核心是**多路竞争**机制：

```
pushToQueue(确认项)   // 推入 UI 确认队列
   |
   +-- 用户在终端操作（onAllow / onReject / onAbort）
   +-- Hook 异步返回结果（runHooks）
   +-- Bash 分类器异步返回（executeAsyncClassifierCheck）
   +-- Bridge 远端响应（CCR / claude.ai）
   +-- Channel 响应（Telegram / iMessage）
```

使用 `createResolveOnce` + `claim()` 保证**只有第一个响应生效**，其余被丢弃。

关键特性：
- 200ms 宽限期：防止用户意外按键取消分类器
- 分类器自动通过后显示 checkmark，3 秒后消失（终端不聚焦时 1 秒）
- 支持 Esc 提前关闭 checkmark

### 5.2 协调器处理器（`coordinatorHandler.ts`）

用于协调器工作者（coordinator worker）。与交互处理器的区别：**先串行跑完自动化检查，再回退到对话框**。

```
1. 运行 Hook -> 如果有决定则返回
2. 运行分类器 -> 如果有决定则返回
3. 都没结果 -> 返回 null，调用方回退到交互对话框
```

### 5.3 Swarm 工作者处理器（`swarmWorkerHandler.ts`）

Swarm 多 agent 场景中，工作者进程的权限请求会**转发给 Leader**：

```
1. 尝试分类器自动通过
2. 通过 mailbox 发送权限请求给 Leader
3. 注册回调等待 Leader 响应
4. 显示 pendingWorkerRequest 等待指示器
5. Leader 响应 -> 通过回调 resolve
```

## 六、PermissionContext（权限上下文对象）

`src/hooks/toolPermission/PermissionContext.ts` 定义了统一的权限操作上下文，每次工具调用创建一个：

| 方法 | 作用 |
|------|------|
| `logDecision()` | 记录权限决定（分析 + 遥测） |
| `persistPermissions()` | 持久化权限规则更新到磁盘 |
| `resolveIfAborted()` | 检查是否已中止 |
| `cancelAndAbort()` | 构造拒绝决定并中止控制器 |
| `runHooks()` | 执行 PermissionRequest hook |
| `tryClassifier()` | 尝试 Bash 分类器自动通过 |
| `handleUserAllow()` | 处理用户允许（持久化 + 日志） |
| `handleHookAllow()` | 处理 Hook 允许 |
| `buildAllow() / buildDeny()` | 构建标准化的决定对象 |
| `pushToQueue() / removeFromQueue()` | 操作 UI 确认队列 |

## 七、权限日志与遥测（`permissionLogging.ts`）

所有权限决定统一通过 `logPermissionDecision()` 记录：

- **Analytics 事件** -- 区分 granted_in_config / granted_by_classifier / granted_in_prompt_permanent 等
- **OTel 遥测** -- tool_decision 事件
- **代码编辑计数** -- Edit/Write/NotebookEdit 工具的 accept/reject 计数器
- **ToolUseContext 存储** -- `toolDecisions` Map，供下游检查

## 八、关键数据流总结

```
用户提示 -> LLM 返回 tool_use -> useCanUseTool()
  |
  v
hasPermissionsToUseTool()
  |-- allow rules 匹配? -> allow
  |-- deny rules 匹配?  -> deny
  |-- 工具 checkPermissions() -> allow/deny/ask
  |
  v (ask)
权限模式后处理
  |-- dontAsk -> deny
  |-- auto -> AI 分类器 -> allow/deny
  |-- default/plan/acceptEdits -> ask
  |
  v (ask)
Handler 多路竞争
  |-- 用户终端交互
  |-- PermissionRequest Hook
  |-- Bash 分类器
  |-- Bridge (claude.ai)
  |-- Channel (Telegram/iMessage)
  |
  v (第一个响应胜出)
PermissionDecision { behavior, updatedInput, decisionReason }
  |
  v
工具执行 / 拒绝反馈
```

## 九、设计要点

1. **单一状态树** -- 所有状态集中在 AppState，通过 selector 订阅切片
2. **副作用集中** -- onChangeAppState 统一处理模式同步、配置持久化
3. **竞争模型** -- 权限决定采用 claim() 原子操作，多路 resolve 只有首个生效
4. **分层判定** -- 规则优先、模式次之、交互兜底
5. **可扩展 Hook** -- PermissionRequest hook 可在任何环节介入
6. **拒绝追踪** -- auto 模式下连续拒绝超限后回退到交互提示
