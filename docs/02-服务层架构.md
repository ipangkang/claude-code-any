# Claude Code 服务层架构

> 基于 `src/services/` 目录（约 130 个文件）的源码分析。

---

## 一、整体组织

服务层采用 **按领域分目录** 的扁平结构，无统一基类或注册表：

```
src/services/
├── api/            # API 客户端与请求管理（14 文件）
├── mcp/            # MCP 协议集成（20+ 文件）
├── compact/        # 上下文压缩（10 文件）
├── tools/          # 工具执行引擎（4 文件）
├── lsp/            # 语言服务协议（7 文件）
├── analytics/      # 遥测与事件上报（8 文件）
├── SessionMemory/  # 会话记忆
├── compact/        # 上下文压缩
├── plugins/        # 插件管理
└── ...             # 其余独立服务文件
```

**设计特点：**
- 无依赖注入框架，通过直接 import 组合
- 大量使用 `feature()` (来自 `bun:bundle`) 做编译期特性开关，实现死代码消除
- 服务以函数/工厂函数为主，少用 class（LSP 和 StreamingToolExecutor 例外）

## 二、API 调用流程 (`src/services/api/`)

### 2.1 客户端创建 — `client.ts`

`getAnthropicClient()` 是统一入口，支持四种后端：

| 后端 | 环境变量 | SDK |
|------|---------|-----|
| Anthropic 直连 | `ANTHROPIC_API_KEY` | `@anthropic-ai/sdk` |
| AWS Bedrock | `CLAUDE_CODE_USE_BEDROCK` | `@anthropic-ai/bedrock-sdk` |
| Azure Foundry | `CLAUDE_CODE_USE_FOUNDRY` | `@anthropic-ai/foundry-sdk` |
| GCP Vertex | `CLAUDE_CODE_USE_VERTEX` | `@anthropic-ai/vertex-sdk` |

关键细节：
- 默认超时 600 秒，可通过 `API_TIMEOUT_MS` 覆盖
- 自动注入 `x-app: cli`、Session ID、User-Agent 等请求头
- OAuth 令牌自动刷新（`checkAndRefreshOAuthTokenIfNeeded`）

### 2.2 请求发送 — `claude.ts`

核心函数 `queryModelWithStreaming()`，职责：
1. 消息规范化（`normalizeMessagesForAPI`）
2. 工具 schema 转换（`toolToAPISchema`）
3. 注入 beta 头（思考模式、缓存、Effort 等级等）
4. 流式响应处理，逐块 yield `StreamEvent`
5. 成本追踪（`addToTotalSessionCost`）

### 2.3 重试机制 — `withRetry.ts`

```
重试策略:
- 基础延迟 500ms，指数退避
- 默认最多 10 次重试
- 529（过载）：仅前台请求重试，最多 3 次
- 429/529 持久重试：无人值守模式下无限重试，最大退避 5 分钟
- 401：自动刷新 OAuth/AWS/GCP 凭据后重试
```

### 2.4 引导数据 — `bootstrap.ts`

启动时调用 `/api/claude_cli/bootstrap` 获取：
- `client_data`：动态配置
- `additional_model_options`：额外可用模型列表
- 结果缓存到磁盘，仅变更时写入

## 三、MCP 协议集成 (`src/services/mcp/`)

### 3.1 客户端 — `client.ts`

基于 `@modelcontextprotocol/sdk`，支持多种传输：

| 传输方式 | 类 |
|---------|---|
| Stdio 子进程 | `StdioClientTransport` |
| SSE | `SSEClientTransport` |
| Streamable HTTP | `StreamableHTTPClientTransport` |
| WebSocket | `WebSocketTransport` |
| SDK 控制通道 | `SdkControlClientTransport` |
| 进程内 | `InProcessTransport` |

**连接生命周期：**
1. 从配置读取 MCP 服务器定义（`getAllMcpConfigs`）
2. 创建传输层，建立连接
3. 发现工具 → 包装为 `MCPTool` 实例
4. 发现资源 → 包装为 `ReadMcpResourceTool` / `ListMcpResourcesTool`

**错误处理：**
- `McpAuthError`：OAuth 过期 → 标记服务器为 `needs-auth`
- `McpSessionExpiredError`：会话 404 → 清缓存重连
- `McpToolCallError`：工具返回 `isError: true`

### 3.2 连接管理 — `MCPConnectionManager.tsx`

React Context 模式，提供：
- `reconnectMcpServer(name)` — 重连指定服务器
- `toggleMcpServer(name)` — 启用/禁用服务器

通过 `useManageMCPConnections` hook 管理所有连接状态。

## 四、上下文压缩策略 (`src/services/compact/`)

### 4.1 三层压缩体系

```
┌─────────────────────────────────────────┐
│  MicroCompact（工具结果级，每轮触发）       │  ← 最细粒度
├─────────────────────────────────────────┤
│  AutoCompact（自动全量摘要）               │  ← token 阈值触发
├─────────────────────────────────────────┤
│  Manual Compact（用户手动 /compact）       │  ← 用户触发
└─────────────────────────────────────────┘
```

### 4.2 MicroCompact — `microCompact.ts`

- 仅压缩特定工具结果：FileRead、Bash、Grep、Glob、WebSearch 等
- 基于时间策略（`timeBasedMCConfig`），老旧结果标记为 `[Old tool result content cleared]`
- 支持缓存编辑模式（`cachedMicrocompact`），通过 `cache_edits` 块优化 API 缓存命中

### 4.3 AutoCompact — `autoCompact.ts`

触发条件：`token 使用量 >= 上下文窗口 - 输出预留 - 13K 缓冲`

```typescript
// 关键阈值
AUTOCOMPACT_BUFFER_TOKENS    = 13_000   // 触发缓冲
WARNING_THRESHOLD_BUFFER_TOKENS = 20_000 // 警告阈值
MAX_OUTPUT_TOKENS_FOR_SUMMARY  = 20_000  // 摘要输出预留
```

- 连续失败 3 次后停止重试（断路器）
- 可通过 `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` 按百分比覆盖阈值

### 4.4 全量压缩 — `compact.ts`

**压缩流程：**
1. 剥离图片/文档（`stripImagesFromMessages`）→ 避免摘要请求超限
2. 剥离可重注入的附件（skill 发现等）
3. 调用模型生成摘要（使用专用 prompt）
4. 创建 `SystemCompactBoundaryMessage` 分界点
5. 后处理（`postCompactCleanup`）：恢复最近编辑的文件内容（最多 5 个，50K token）

**Prompt 设计（`prompt.ts`）：**
- 9 节结构化摘要：主要意图、技术概念、文件变更、错误修复、用户消息等
- 强制 `<analysis>` 草稿块 + `<summary>` 最终块
- 严格禁止工具调用（`NO_TOOLS_PREAMBLE`）

**容错：**
- Prompt Too Long → 逐步裁剪最老的消息组，最多重试 3 次
- 用户中止 → 优雅退出

## 五、工具执行引擎 (`src/services/tools/`)

### 5.1 编排 — `toolOrchestration.ts`

`runTools()` 将工具调用分为两类批次：

```
并发安全（只读）工具 → 并行执行（最大并发 10）
非并发安全工具       → 串行执行
```

分批逻辑（`partitionToolCalls`）：连续的并发安全工具合并为一个批次。

### 5.2 执行 — `toolExecution.ts`

`runToolUse()` 单工具执行流程：
1. 权限检查 → Hook 前置执行（`runPreToolUseHooks`）
2. 工具执行 + 流式进度上报
3. Hook 后置执行（`runPostToolUseHooks`）
4. 错误分类（`classifyToolError`）→ 遥测安全字符串

### 5.3 流式执行器 — `StreamingToolExecutor`

唯一使用 class 的服务，管理流式场景下的工具调度：
- 工具在流入时即开始执行（不等流结束）
- 结果按接收顺序缓冲输出
- `discard()` 支持流式回退时丢弃中间结果

## 六、语言服务 (`src/services/lsp/`)

### 架构

```
LSPServerManager（路由层）
  ├── LSPServerInstance（单服务器实例）
  │     ├── start/stop 生命周期
  │     ├── stdio 通信
  │     └── 文件同步（didOpen/didChange/didSave/didClose）
  └── 文件扩展名 → 服务器映射
```

**工厂函数模式**（非 class）：`createLSPServerManager()` 返回闭包封装的接口。

核心能力：
- 按文件扩展名路由到对应 LSP 服务器
- 支持 `textDocument/definition` 等标准请求
- `LSPDiagnosticRegistry` 收集诊断信息供 AI 参考

## 七、遥测系统 (`src/services/analytics/`)

### 设计

```
logEvent() / logEventAsync()
    ↓ （启动前排队）
attachAnalyticsSink()
    ↓
AnalyticsSink
    ├── Datadog（通用指标）
    └── 1P Event Logger（第一方日志）
```

**隐私保护：**
- 元数据类型 `AnalyticsMetadata_I_VERIFIED_THIS_IS_NOT_CODE_OR_FILEPATHS` 强制开发者确认无敏感信息
- `_PROTO_*` 前缀字段仅发送到 PII 标记列，`stripProtoFields()` 在通用存储前过滤
- GrowthBook 集成（`growthbook.ts`）用于特性开关和 A/B 测试

**启动时序：**
1. 事件写入队列
2. `attachAnalyticsSink()` 注入后端（幂等）
3. 队列通过 `queueMicrotask` 异步排空

## 八、服务生命周期总结

| 阶段 | 关键动作 |
|------|---------|
| 引导 | `bootstrap.ts` 拉取配置；`analytics` 排队事件 |
| 初始化 | API 客户端创建；MCP 服务器连接；LSP 启动 |
| 运行 | 工具执行（编排+并发）；MicroCompact 持续清理 |
| 压缩 | AutoCompact 阈值触发 → 全量摘要 → 后处理恢复 |
| 关闭 | LSP shutdown；MCP 断连；Analytics flush |

**跨服务依赖关系：**
```
tools/ ──→ mcp/client（MCP 工具调用）
compact/ ──→ api/claude（调用模型生成摘要）
api/claude ──→ compact/microCompact（请求前压缩工具结果）
analytics/ ←── 所有服务（事件上报）
```
