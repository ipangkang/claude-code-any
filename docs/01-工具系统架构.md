# Claude Code 工具系统架构

## 一、工具总览

Claude Code 的工具系统由 `src/Tool.ts` 定义核心接口，`src/tools.ts` 负责注册和组装，各工具实现在 `src/tools/` 子目录下。

## 二、核心接口 `Tool<Input, Output, P>`

定义在 `src/Tool.ts`，约 40+ 个方法/属性，关键接口：

```typescript
type Tool<Input, Output, P> = {
  name: string;
  aliases?: string[];
  inputSchema: ZodSchema;          // Zod v4 校验
  outputSchema?: ZodSchema;

  // 生命周期
  call(args, context, canUseTool, parentMessage, onProgress?): Promise<ToolResult>;
  validateInput?(input, context): Promise<ValidationResult>;
  checkPermissions(input, context): Promise<PermissionResult>;

  // 元信息
  description(input, options): Promise<string>;
  prompt(options): Promise<string>;  // 给模型看的说明
  isEnabled(): boolean;
  isConcurrencySafe(input): boolean;
  isReadOnly(input): boolean;

  // UI 渲染（React/Ink）
  renderToolUseMessage(input, options): ReactNode;
  renderToolResultMessage?(content, progress, options): ReactNode;
  renderToolUseErrorMessage?(result, options): ReactNode;

  // 权限匹配
  preparePermissionMatcher?(input): Promise<(pattern: string) => boolean>;
  getPath?(input): string;

  // 结果序列化
  mapToolResultToToolResultBlockParam(content, toolUseID): ToolResultBlockParam;
  maxResultSizeChars: number;       // 超过则持久化到磁盘
}
```

### `buildTool(def)` 工厂函数

所有工具通过 `buildTool()` 构建，自动填充默认值：
- `isEnabled` -> `true`
- `isConcurrencySafe` -> `false`（保守策略）
- `isReadOnly` -> `false`（假设有写操作）
- `checkPermissions` -> `allow`（交给通用权限系统）

## 三、工具注册与调度

### 注册流程 (`src/tools.ts`)

```
getAllBaseTools()        -- 返回所有内置工具数组（含条件加载）
    |
getTools(permCtx)       -- 按权限过滤 + deny规则 + 模式过滤
    |
assembleToolPool(permCtx, mcpTools)  -- 合并 MCP 工具，按名称去重
```

关键设计：
1. **条件加载**：通过 `feature()` (Bun bundle-time) 和环境变量实现死代码消除
2. **deny 规则过滤**：`filterToolsByDenyRules()` 在模型看到工具列表前移除被禁工具
3. **MCP 工具合并**：内置工具优先，按名称去重，分区排序保证 prompt cache 稳定
4. **Simple 模式**：`CLAUDE_CODE_SIMPLE=true` 时只暴露 Bash/Read/Edit
5. **REPL 模式**：将原始工具包装在 VM 内部，对外只暴露 REPLTool

### 工具调用链

```
模型输出 tool_use -> findToolByName(tools, name) -> validateInput()
    -> checkPermissions() -> call() -> mapToolResultToToolResultBlockParam()
```

## 四、权限系统

### 三层权限检查

1. **deny 规则预过滤**：工具注册时即移除被全局 deny 的工具
2. **validateInput()**：工具内部校验（文件是否存在、是否已读、路径安全等）
3. **checkPermissions()**：工具级权限判定，返回 `PermissionResult`

### PermissionResult 行为

```typescript
type PermissionResult =
  | { behavior: 'allow'; updatedInput }   // 允许
  | { behavior: 'ask'; ... }              // 需要用户确认
  | { behavior: 'deny'; message }         // 拒绝
```

### Bash 权限（最复杂）

`BashTool/bashPermissions.ts` 实现多层安全：
- **AST 解析**：`parseForSecurity()` 用 tree-sitter 解析命令
- **通配符规则匹配**：`matchWildcardPattern()` 支持 `git *` 等模式
- **分类器**：`classifyBashCommand()` 用 AI 分类器判断命令安全性
- **沙箱**：`shouldUseSandbox()` 判断是否需要沙箱隔离
- **只读验证**：`checkReadOnlyConstraints()` 在计划模式下限制写操作
- **路径验证**：`checkPathConstraints()` 防止 UNC 路径 NTLM 泄露
- **sed 验证**：`checkSedConstraints()` 专门处理 sed 编辑命令

### 文件系统权限

`utils/permissions/filesystem.ts` 提供：
- `checkWritePermissionForTool()` -- FileEdit/FileWrite 使用
- `checkReadPermissionForTool()` -- Grep/Glob/FileRead 使用
- `matchingRuleForInput()` -- 匹配 deny/allow/ask 规则

### ToolPermissionContext

```typescript
type ToolPermissionContext = {
  mode: PermissionMode;          // 'default' | 'plan' | ...
  alwaysAllowRules: {...};       // 按来源分组的允许规则
  alwaysDenyRules: {...};        // 按来源分组的拒绝规则
  alwaysAskRules: {...};         // 按来源分组的询问规则
  additionalWorkingDirectories;  // 额外允许的工作目录
  shouldAvoidPermissionPrompts?; // 后台 agent 不弹权限对话框
}
```

## 五、AgentTool：子 Agent 实现

### 架构

AgentTool 是最复杂的工具，支持三种运行模式：

1. **同步子 agent**：在当前进程内运行，阻塞等待结果
2. **后台异步 agent**：`run_in_background=true`，注册为 LocalAgentTask
3. **Teammate（多 agent 协作）**：通过 tmux/进程内方式 spawn 独立 agent

### 核心流程

```
AgentTool.call()
  |-- 检查 subagent_type（内置/自定义 agent 类型）
  |-- 解析 model（sonnet/opus/haiku）
  |-- 检查 isolation（worktree/remote）
  |-- 如果是 teammate: spawnTeammate() -> 返回
  |-- 如果是 remote: teleportToRemote() -> 返回
  |-- runAgent() 核心执行：
        |-- createSubagentContext() 创建隔离的上下文
        |-- 克隆 readFileState（文件状态隔离）
        |-- 组装子工具池（assembleToolPool）
        |-- 构建 system prompt
        |-- query() 调用 Claude API
        |-- 收集结果 -> 返回给父 agent
```

### 内置 Agent 类型 (`AgentTool/built-in/`)

- `generalPurposeAgent` -- 通用 agent（默认）
- `exploreAgent` -- 代码探索
- `planAgent` -- 规划
- `verificationAgent` -- 验证
- `claudeCodeGuideAgent` -- Claude Code 使用指导

### Agent 定义 (`AgentDefinition`)

自定义 agent 通过 `.claude/agents/` 目录加载，支持 frontmatter 定义：
- `model`: 指定模型
- `mcpServers`: agent 专属 MCP 服务器
- `color`: UI 颜色
- `source`: 来源（user/plugin/built-in）

### 子 Agent 上下文隔离

`createSubagentContext()` (`utils/forkedAgent.ts`) 为子 agent 创建隔离环境：
- 独立的 `readFileState` 缓存
- 独立的 `AbortController`
- 共享的 `setAppStateForTasks`（任务管理穿透到根）
- 独立的 session hooks

## 六、共享逻辑 (`src/tools/shared/`)

### `spawnMultiAgent.ts`

Teammate（多 agent）的 spawn 实现：
- 支持 tmux 后端（独立终端）和进程内后端
- 自动检测可用后端（tmux > in-process fallback）
- 构建启动命令（`claude` CLI 参数）
- 管理 team roster 和 agent 注册

### `gitOperationTracking.ts`

Shell 命令中的 git 操作检测：
- 正则匹配 `git commit/push/cherry-pick/merge/rebase`
- 检测 `gh pr create/edit/merge/close`
- 检测 curl 方式创建 PR
- 触发 OTLP 计数器和 analytics 事件
- 自动关联 session 到 PR

### `src/tools/utils.ts`

消息标记工具：
- `tagMessagesWithToolUseID()` -- 关联 user 消息与 tool use
- `getToolUseIDFromParentMessage()` -- 从 assistant 消息提取 tool use ID

## 七、工具搜索与延迟加载

当工具数量超过阈值时，启用 `ToolSearchTool`：
- 部分工具以 `shouldDefer=true` 标记，不在初始 prompt 中展开完整 schema
- 模型通过 `ToolSearchTool` 按关键字搜索工具
- `searchHint` 属性提供搜索关键词（如 GrepTool: "search file contents with regex"）
- `alwaysLoad=true` 的工具始终展开（不延迟）

## 八、设计要点总结

1. **buildTool 工厂模式**：统一构建，默认值 fail-closed
2. **三层权限**：deny 预过滤 -> validateInput -> checkPermissions
3. **Bash 安全最深**：AST 解析 + 分类器 + 沙箱 + 路径校验
4. **子 Agent 隔离**：独立文件状态缓存、独立 abort、共享任务注册
5. **MCP 扩展**：外部 MCP 工具与内置工具统一调度，deny 规则统一生效
6. **条件编译**：`feature()` + 环境变量实现工具的死代码消除
7. **大结果持久化**：超过 `maxResultSizeChars` 的结果存盘，返回预览 + 路径
