# 08 - 启动引导与远程通信

本章覆盖 Claude Code 的全局状态引导（bootstrap）、跨进程桥接（bridge）、远程会话管理（remote）及直连服务器（server）四大子系统。

---

## 1. src/bootstrap/ — 全局启动状态

仅含一个核心文件 `state.ts`，是整个应用的**进程级单例状态容器**。

### 1.1 State 类型

`State` 接口定义了约 80+ 个字段，涵盖：

| 分类 | 典型字段 | 说明 |
|------|---------|------|
| 路径 | `originalCwd`, `projectRoot`, `cwd` | 工作目录（解析符号链接后） |
| 成本与计量 | `totalCostUSD`, `modelUsage`, `tokenCounter` | API 调用统计 |
| 遥测 | `meter`, `loggerProvider`, `tracerProvider` | OpenTelemetry 集成 |
| 会话 | `sessionId`, `parentSessionId` | UUID 标识，支持会话继承 |
| 模型 | `mainLoopModelOverride`, `initialMainLoopModel` | 运行时模型切换 |
| 远程 | `isRemoteMode`, `directConnectServerUrl` | 远程/直连模式标志 |
| 缓存 | `systemPromptSectionCache`, `promptCache1hAllowlist` | 提示缓存优化 |

### 1.2 关键导出

- **`getInitialState()`** — 构造默认状态，解析 `process.cwd()` 并 NFC 规范化
- **`getSessionId()` / `regenerateSessionId()`** — 会话 ID 管理
- **`getOriginalCwd()`** — 被 bridge/remote 子系统用于注册环境时报告工作目录
- **`createSignal()`** — 轻量响应式信号（import 自 `utils/signal.js`）

设计要点：文件顶部注释 `DO NOT ADD MORE STATE HERE`，要求开发者审慎添加全局状态。

## 2. src/bridge/ — 跨进程桥接系统（Remote Control）

Bridge 是 Claude Code 的 **Remote Control** 核心，允许用户通过 claude.ai 网页端远程操控本地 CLI 会话。分为 v1（基于 Environments API 轮询）和 v2（env-less 直连）两代架构。

### 2.1 整体架构

```
claude.ai (Web) ──► CCR 后端 ──► Bridge Worker (本地 CLI)
                                   │
                     ┌─────────────┼─────────────┐
                     v1 (env-based)│  v2 (env-less)
                     register ─►   │  POST /bridge ─►
                     poll ─►       │  SSE + CCRClient
                     ack/heartbeat │
```

### 2.2 核心模块

#### types.ts — 协议类型定义
- `BridgeConfig` — 环境注册参数（目录、分支、Git URL、最大会话数、spawn 模式）
- `WorkResponse` / `WorkSecret` — 轮询返回的任务与加密凭证
- `SessionHandle` — 子进程会话句柄（kill/forceKill/writeStdin/updateAccessToken）
- `BridgeApiClient` — API 客户端接口（register/poll/ack/stop/heartbeat/archive）
- `SpawnMode` — `single-session` | `worktree` | `same-dir`

#### bridgeApi.ts — HTTP API 客户端
- **`createBridgeApiClient(deps)`** — 工厂函数，返回 `BridgeApiClient` 实现
- **`withOAuthRetry()`** — 401 时自动刷新 OAuth token 并重试一次
- **`BridgeFatalError`** — 不可重试错误（401/403/404/410）
- **`validateBridgeId()`** — 防路径注入校验（仅允许 `[a-zA-Z0-9_-]`）

#### bridgeEnabled.ts — 功能门控
- **`isBridgeEnabled()`** — 运行时检查：需 claude.ai 订阅 + GrowthBook 特性标志
- **`isBridgeEnabledBlocking()`** — 阻塞版本，等待 GrowthBook 初始化
- **`getBridgeDisabledReason()`** — 返回可操作的诊断信息
- **`isEnvLessBridgeEnabled()`** — v2 路径门控（`tengu_bridge_repl_v2`）
- **`checkBridgeMinVersion()`** — 最低版本检查

#### bridgeConfig.ts — 认证与 URL 解析
- **`getBridgeAccessToken()`** — 优先使用 dev 环境变量，fallback 到 OAuth keychain
- **`getBridgeBaseUrl()`** — API 基础 URL（dev override 或生产配置）

#### bridgeMain.ts — v1 多会话服务器（`claude remote-control` 命令入口）
- **`runBridgeMain()`** — 注册环境 → 轮询任务 → 派发会话 → 心跳保活
- 支持 `worktree` 模式（每个会话独立 git worktree）和 `same-dir` 模式
- 优雅退出：SIGINT/SIGTERM → kill 所有子进程 → deregister 环境
- 指数退避重连，环境过期自动重新注册

#### replBridge.ts — v1 REPL 内桥接核心
- **`initBridgeCore()`** — 单会话桥接核心（~2400 行），处理：
  - 环境注册与轮询
  - WebSocket/SSE 传输层管理
  - 消息出站（writeMessages/writeSdkMessages）与入站路由
  - 初始历史同步（flush gate 机制）
  - 断线重连与传输层切换（v1 HybridTransport / v2 SSE+CCRClient）

#### remoteBridgeCore.ts — v2 无环境层桥接
- **`initEnvLessBridgeCore()`** — 跳过 Environments API：
  1. `POST /v1/code/sessions` 创建会话
  2. `POST /v1/code/sessions/{id}/bridge` 获取 worker_jwt
  3. 建立 SSE + CCRClient 传输
  4. 定时刷新 JWT（proactive refresh）
  5. 401 时重建传输层

#### replBridgeTransport.ts — 传输层抽象
- **`ReplBridgeTransport`** 接口 — 统一 v1/v2 读写操作
- **`createV1ReplTransport()`** — 包装 HybridTransport（WS 读 + POST 写）
- **`createV2ReplTransport()`** — 包装 SSETransport（读）+ CCRClient（写），自动注册 worker epoch

#### bridgeMessaging.ts — 消息处理工具
- **`handleIngressMessage()`** — 解析入站 WebSocket 消息，路由到相应处理器
- **`handleServerControlRequest()`** — 响应服务端控制请求（initialize/set_model/interrupt 等）
- **`BoundedUUIDSet`** — FIFO 环形缓冲集合，用于消息去重（防回声/重放）
- **`extractTitleText()`** — 从用户消息提取会话标题文本

#### sessionRunner.ts — 子进程管理
- **`createSessionSpawner()`** — 工厂函数，spawn 子 CLI 进程
- 子进程参数：`--print --sdk-url ... --session-id ... --input-format stream-json`
- 监控 stdout NDJSON 提取活动（工具调用、文本输出）
- 检测 `control_request`（权限请求）并上报给 bridge
- 通过 stdin 下发 token 刷新（`update_environment_variables` 消息）

#### initReplBridge.ts — REPL 入口封装
- **`initReplBridge()`** — 汇聚所有引导逻辑：
  1. 检查门控（enabled/version/oauth/policy）
  2. 刷新 OAuth token（防过期）
  3. 跨进程退避（dead token 计数）
  4. 派生会话标题（第 1 条消息占位 → Haiku 生成 → 第 3 条消息精化）
  5. 根据 GrowthBook 门控选择 v1 或 v2 路径

#### 其他辅助模块
| 模块 | 职责 |
|------|------|
| `workSecret.ts` | 解码 base64url 工作凭证，构建 SDK URL |
| `sessionIdCompat.ts` | `cse_*` ↔ `session_*` ID 前缀转换 |
| `jwtUtils.ts` | JWT 定时刷新调度器 |
| `trustedDevice.ts` | 可信设备 token 管理 |
| `flushGate.ts` | 初始历史同步的闸门控制 |
| `capacityWake.ts` | 会话容量变化唤醒信号 |
| `pollConfig.ts` | 轮询间隔动态配置 |
| `createSession.ts` | 通过 `/v1/sessions` 创建桥接会话 |
| `bridgeUI.ts` | 终端 UI 状态显示（idle/attached/reconnecting） |
| `bridgeStatusUtil.ts` | 格式化持续时间等工具函数 |

## 3. src/remote/ — 远程会话客户端

Remote 模块是 **Web 端远程操控的客户端实现**，用于从本地 CLI 连接并操控 CCR 上的远程会话。

### 3.1 RemoteSessionManager.ts — 会话管理器

核心类 `RemoteSessionManager`，协调三个通道：

- **WebSocket 订阅** — 接收 CCR 推送的 SDK 消息
- **HTTP POST** — 向 CCR 发送用户消息（`sendEventToRemoteSession`）
- **权限请求/响应** — 本地审批远程工具调用

关键方法：
- `connect()` — 建立 WebSocket 连接
- `sendMessage(content)` — HTTP POST 发送用户输入
- `respondToPermissionRequest(requestId, result)` — 回复权限决策（allow/deny）
- `cancelSession()` — 发送 interrupt 控制请求

### 3.2 SessionsWebSocket.ts — WebSocket 客户端

连接 `wss://.../v1/sessions/ws/{id}/subscribe`，核心特性：

- 通过 HTTP header 携带 OAuth token 认证
- 自动重连（最多 5 次，2s 间隔）
- 4001（session not found）特殊处理：最多重试 3 次（应对压缩期间的瞬态错误）
- 4003（unauthorized）永久关闭不重连
- 30s 心跳 ping 保活
- 兼容 Bun 原生 WebSocket 和 Node.js `ws` 包

### 3.3 remotePermissionBridge.ts — 权限桥接

- **`createSyntheticAssistantMessage()`** — 为远程权限请求构造合成 AssistantMessage
- **`createToolStub()`** — 为本地未加载的远程工具创建最小 Tool 桩对象

### 3.4 sdkMessageAdapter.ts — 消息格式转换

**`convertSDKMessage(msg)`** — 将 CCR 的 SDK 格式消息转换为 REPL 内部 Message 类型：

| SDK 类型 | 转换结果 |
|---------|---------|
| `assistant` | `AssistantMessage` |
| `stream_event` | `StreamEvent` |
| `result`（错误时） | `SystemMessage` |
| `system.init` | `SystemMessage`（显示模型信息） |
| `system.status` | `SystemMessage`（如 "Compacting..."） |
| `system.compact_boundary` | `SystemMessage`（压缩边界） |
| `tool_progress` | `SystemMessage`（工具进度） |
| `user`（默认） | 忽略（REPL 已本地添加） |

## 4. src/server/ — 直连服务器

Server 模块实现 **Direct Connect**，允许 CLI 直接连接到自托管的 Claude 服务器，无需经过 CCR 云端。

### 4.1 types.ts — 服务器类型

- `ServerConfig` — 服务器配置（port/host/authToken/unix/idleTimeoutMs/maxSessions）
- `SessionState` — 会话状态机：`starting → running → detached → stopping → stopped`
- `SessionInfo` — 会话元数据（ID、状态、进程句柄、工作目录）
- `SessionIndexEntry` / `SessionIndex` — 持久化索引（`~/.claude/server-sessions.json`），支持跨重启恢复
- `connectResponseSchema` — Zod 校验：`{ session_id, ws_url, work_dir? }`

### 4.2 createDirectConnectSession.ts — 会话创建

**`createDirectConnectSession({ serverUrl, authToken, cwd })`**：
- POST `${serverUrl}/sessions` 创建会话
- 返回 `DirectConnectConfig`（serverUrl + sessionId + wsUrl）
- 自定义错误类 `DirectConnectError`

### 4.3 directConnectManager.ts — WebSocket 会话管理

**`DirectConnectSessionManager`** 类，功能类似 `RemoteSessionManager` 但面向直连场景：

- WebSocket 连接，通过 Authorization header 认证
- 解析 NDJSON 消息流，分发 SDK 消息和控制请求
- `sendMessage(content)` — 构造 `SDKUserMessage` 格式发送
- `respondToPermissionRequest()` — 回复权限决策
- `sendInterrupt()` — 发送中断控制请求
- 过滤 `keep_alive`、`control_response`、`streamlined_*` 等非显示消息

## 架构总结

```
用户本地 CLI
  ├── bootstrap/state.ts ─── 全局单例状态（启动时初始化）
  │
  ├── bridge/ ─── Remote Control 工作端（接受 claude.ai 控制）
  │   ├── v1: register → poll → ack → spawn child → heartbeat
  │   └── v2: POST /bridge → SSE+CCRClient（无轮询）
  │
  ├── remote/ ─── 远程操控客户端（连接并控制远程会话）
  │   └── WebSocket subscribe + HTTP POST + 权限审批
  │
  └── server/ ─── Direct Connect（自托管，无云端中转）
      └── POST /sessions → WebSocket NDJSON 双向通信
```

关键设计决策：
1. **传输层抽象**（`ReplBridgeTransport`）使 v1/v2 切换对上层透明
2. **OAuth 弹性**：过期检测 → 自动刷新 → 跨进程退避 → 死 token 计数
3. **消息去重**：`BoundedUUIDSet` 环形缓冲防止回声和重放
4. **渐进式标题**：第 1 条消息截取占位 → Haiku 模型生成 → 第 3 条精化
