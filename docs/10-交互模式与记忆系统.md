# 10 - 交互模式与记忆系统

本章涵盖 Claude Code 的六个子系统：Vim 模式、语音模式、快捷键系统、输出样式、记忆目录（auto memory）和迁移系统。

---

## 1. Vim 模式 (`src/vim/`)

### 架构概述

Vim 模式是一个**纯函数式状态机**，将终端键盘输入映射为文本编辑操作。核心设计：类型即文档，状态机即逻辑。

### 文件清单

| 文件 | 职责 |
|------|------|
| `types.ts` | 状态机类型定义、按键常量集合、工厂函数 |
| `motions.ts` | 光标移动的纯计算函数 (`h/j/k/l/w/b/e/$` 等) |
| `operators.ts` | 操作符执行 (`delete/change/yank`) 及相关命令 (`x/r/~/J/p/o/>>`) |
| `textObjects.ts` | 文本对象查找 (`iw/aw/i"/a(` 等) |
| `transitions.ts` | 状态转移表 —— 所有按键到状态的映射 |

### 状态机设计

```
VimState = INSERT { insertedText } | NORMAL { CommandState }

CommandState 的子状态:
  idle → count | operator | find | g | replace | indent
  operator → operatorCount | operatorFind | operatorTextObj | operatorG
```

- **INSERT 模式**：记录 `insertedText` 用于 dot-repeat
- **NORMAL 模式**：`CommandState` 联合类型，TypeScript 确保 switch 穷举
- **PersistentState**：跨命令持久化 —— `lastChange`(dot-repeat)、`lastFind`(`;/,` 重复)、`register`(粘贴板)

### 关键实现

- `resolveMotion()`: 循环 count 次调用 `applySingleMotion()`，遇到边界提前退出
- `executeOperatorMotion()`: 计算目标位置 → 判断 inclusive/linewise → 调用 `applyOperator()`
- `transition()`: 根据 `state.type` 派发到对应的 `fromXxx()` 函数，返回 `{ next?, execute? }`
- `cw/cW` 特殊处理：change 时移动到词尾而非下一词首
- 最大计数 `MAX_VIM_COUNT = 10000` 防止恶意输入

## 2. 语音模式 (`src/voice/`)

### 唯一文件：`voiceModeEnabled.ts`

语音模式依赖 Anthropic OAuth + GrowthBook 功能开关双重门控。

### 三层检查函数

| 函数 | 用途 | 检查内容 |
|------|------|----------|
| `isVoiceGrowthBookEnabled()` | UI 可见性 | GrowthBook `tengu_amber_quartz_disabled` 未开启 + feature('VOICE_MODE') |
| `hasVoiceAuth()` | 认证检查 | `isAnthropicAuthEnabled()` + OAuth accessToken 存在 |
| `isVoiceModeEnabled()` | 运行时完整检查 | 两者的交集 |

- 语音通过 claude.ai 的 `voice_stream` 端点，仅支持 Anthropic OAuth，不支持 API Key / Bedrock / Vertex / Foundry
- 默认快捷键为 `space`（按住说话），配置在 `defaultBindings.ts` 的 Chat 上下文中

## 3. 快捷键系统 (`src/keybindings/`)

### 架构分层

```
用户按键 → parser 解析 → match 匹配 → resolver 解析(含 chord) → useKeybinding 派发 handler
                                                     ↑
                                        loadUserBindings 加载配置
                                        defaultBindings 提供默认值
```

### 核心文件

| 文件 | 职责 |
|------|------|
| `schema.ts` | Zod Schema —— 18 个上下文 + ~80 个动作标识符 |
| `parser.ts` | 将 `"ctrl+shift+k"` 解析为 `ParsedKeystroke`，支持 chord (`"ctrl+x ctrl+k"`) |
| `match.ts` | 将 Ink 的 `Key` 对象与 `ParsedKeystroke` 匹配，处理 alt/meta 合并 |
| `resolver.ts` | 单键解析 (`resolveKey`) + chord 解析 (`resolveKeyWithChordState`) |
| `defaultBindings.ts` | 所有默认绑定，按 context 分组 |
| `loadUserBindings.ts` | 从 `~/.claude/keybindings.json` 加载用户配置，支持热重载 (chokidar) |
| `validate.ts` | 验证用户配置：结构校验、重复检测、保留快捷键冲突检查 |
| `reservedShortcuts.ts` | 不可重绑定的键 (`ctrl+c/d/m`) + 平台保留键 |
| `useKeybinding.ts` | React Hook，组件级绑定 action → handler |
| `KeybindingContext.tsx` | React Context Provider，管理 active contexts + handler 注册 |
| `KeybindingProviderSetup.tsx` | 顶层 Provider 组装，含 chord 超时 (1000ms) + 热重载订阅 |

### 上下文优先级

18 个上下文 (Global, Chat, Autocomplete, Confirmation, Settings, Transcript ...)：
- 用户绑定 append 在默认绑定之后，**后者优先**（last-wins 策略）
- `useRegisterKeybindingContext()` 让组件挂载时注册活跃上下文，卸载时移除
- Chord 序列如 `ctrl+x ctrl+k` 有 1 秒超时

### 用户自定义

格式：`{ "bindings": [ { "context": "Chat", "bindings": { "ctrl+k": "chat:submit" } } ] }`
- 目前通过 GrowthBook `tengu_keybinding_customization_release` 门控
- 支持 `null` 取消默认绑定，支持 `command:xxx` 触发斜杠命令

## 4. 输出样式 (`src/outputStyles/`)

### 唯一文件：`loadOutputStylesDir.ts`

从 Markdown 文件加载自定义输出样式：

- **项目级**：`.claude/output-styles/*.md`
- **用户级**：`~/.claude/output-styles/*.md`（被项目级覆盖）

每个 `.md` 文件的结构：
- frontmatter 中的 `name`、`description`、`keep-coding-instructions` (bool)
- 正文内容作为 style prompt

结果缓存在 `memoize` 中，`clearOutputStyleCaches()` 清除缓存。

## 5. 记忆目录系统 (`src/memdir/`)

### 架构概述

Auto Memory 是 Claude Code 的**持久化文件记忆系统**，让 AI 在跨会话间保持对用户、项目、反馈的记忆。

### 文件清单

| 文件 | 职责 |
|------|------|
| `paths.ts` | 路径解析 + 功能开关 (`isAutoMemoryEnabled`, `getAutoMemPath`) |
| `memoryTypes.ts` | 四种记忆类型定义 + 系统提示词模板 |
| `memdir.ts` | 核心逻辑：提示词构建、MEMORY.md 截断、目录确保 |
| `memoryScan.ts` | 扫描 `.md` 文件，解析 frontmatter，返回 MemoryHeader 列表 |
| `memoryAge.ts` | 记忆新鲜度计算和过期提示 |
| `findRelevantMemories.ts` | 用 Sonnet 侧查询选择与当前查询相关的记忆 (最多 5 条) |
| `teamMemPaths.ts` | 团队记忆路径 + 安全验证 (防路径穿越/符号链接攻击) |
| `teamMemPrompts.ts` | 团队记忆模式的合并提示词构建 |

### 四种记忆类型

| 类型 | 作用域 | 用途 |
|------|--------|------|
| `user` | 始终私有 | 用户角色、偏好、知识背景 |
| `feedback` | 默认私有 | 用户对 AI 行为的纠正/确认 |
| `project` | 偏向团队 | 项目进展、截止日期、决策背景 |
| `reference` | 通常团队 | 外部系统的指针 (Linear, Grafana 等) |

### 路径解析链

```
CLAUDE_COWORK_MEMORY_PATH_OVERRIDE (env)
  → autoMemoryDirectory (settings.json, 仅 policy/local/user)
    → ~/.claude/projects/<sanitized-git-root>/memory/
```

- `getAutoMemPath()` 使用 `memoize`，按 projectRoot 缓存
- 团队记忆路径 = `getAutoMemPath() + "team/"`
- Git worktree 共享同一个记忆目录 (`findCanonicalGitRoot`)

### 安全机制

- `validateMemoryPath()`: 拒绝相对路径、根路径、UNC 路径、null 字节
- `teamMemPaths.ts` 中的 `validateTeamMemWritePath()`: 双重验证
  1. 字符串级别 `resolve()` 消除 `..` 段
  2. 文件系统级别 `realpath()` 解析符号链接，防止 symlink 逃逸
- `sanitizePathKey()`: 拒绝 null 字节、URL 编码穿越、Unicode 正规化攻击

### 记忆召回流程

1. `scanMemoryFiles()` 扫描 `.md` 文件 (排除 MEMORY.md)，最多 200 个
2. `findRelevantMemories()` 将文件清单发给 Sonnet，选出最多 5 个相关文件
3. `memoryAge()` 为超过 1 天的记忆添加过期警告文本
4. MEMORY.md 作为索引文件，限制 200 行 / 25KB，超限截断并警告

### KAIROS 助手模式

长时运行会话使用**日志模式**而非索引模式：
- 追加写入 `logs/YYYY/MM/YYYY-MM-DD.md`
- 夜间 `/dream` 技能将日志蒸馏为主题文件 + MEMORY.md

## 6. 迁移系统 (`src/migrations/`)

### 设计模式

每个迁移是一个**幂等函数**，在启动时执行一次。通过 globalConfig 中的标记位或 settings 内容本身判断是否需要执行。

### 迁移清单

| 文件 | 作用 |
|------|------|
| `migrateAutoUpdatesToSettings.ts` | 将 globalConfig 中用户手动关闭的 `autoUpdates` 迁移为 settings.json 的 `DISABLE_AUTOUPDATER` 环境变量 |
| `migrateBypassPermissionsAcceptedToSettings.ts` | 权限绕过设置迁移 |
| `migrateEnableAllProjectMcpServersToSettings.ts` | MCP 服务器启用设置迁移 |
| `migrateFennecToOpus.ts` | 内部模型别名迁移：fennec-latest → opus, fennec-fast → opus[1m] |
| `migrateLegacyOpusToCurrent.ts` | 旧版 Opus 别名迁移 |
| `migrateOpusToOpus1m.ts` | Opus → Opus[1m] 合并（Max/Team Premium 用户） |
| `migrateReplBridgeEnabledToRemoteControlAtStartup.ts` | REPL bridge 设置迁移 |
| `migrateSonnet1mToSonnet45.ts` | Sonnet[1m] → Sonnet 4.5 迁移 |
| `migrateSonnet45ToSonnet46.ts` | Sonnet 4.5 → Sonnet 4.6 别名迁移 (Pro/Max/Team Premium) |
| `resetAutoModeOptInForDefaultOffer.ts` | auto mode opt-in 重置 |
| `resetProToOpusDefault.ts` | Pro 用户默认模型重置为 Opus 4.5 |

### 典型迁移模式

```typescript
export function migrateSonnet45ToSonnet46(): void {
  // 1. 前提条件检查 (API provider, 订阅类型)
  if (getAPIProvider() !== 'firstParty') return
  if (!isProSubscriber() && !isMaxSubscriber()) return

  // 2. 仅读 userSettings (不读合并后的 settings)
  const model = getSettingsForSource('userSettings')?.model
  if (model !== 'claude-sonnet-4-5-20250929') return

  // 3. 执行迁移
  updateSettingsForSource('userSettings', { model: 'sonnet' })

  // 4. 设置通知标记 (跳过新用户)
  if (config.numStartups > 1) {
    saveGlobalConfig(c => ({ ...c, migrationTimestamp: Date.now() }))
  }

  // 5. 遥测上报
  logEvent('tengu_sonnet45_to_46_migration', { from_model: model })
}
```

关键原则：
- 只读 `userSettings` 源，不读合并配置（避免误迁移项目级设置）
- 幂等：条件不满足则直接 return
- 迁移后通过 `globalConfig` 标记位防止重复执行
- 遥测记录迁移动作，便于监控
